# Commit format (mvp-scope-implementer)

Use when this skill creates git commits on behalf of the user.

## Title

- One line, imperative mood, describes the change and intent.
- Examples:
  - `Add Category model and budget category API`
  - `Validate transaction currency matches account`
  - `Add holdings export page and API client`

## Body (optional)

Use when the step touches multiple areas or non-obvious behavior:

```text
Short title

- Bullet: what changed
- Bullet: tests or docs updated
```

Skip the body when the title alone is sufficient for a small, single-file change.

## Forbidden in commits

Never include references to tooling or authorship such as:

- Cursor, AI, Claude, GPT, Copilot, agent, assistant
- "generated by", "co-authored-by" automation bots
- Plan file names or internal skill names

Write as the repository owner would: factual, professional, no meta-commentary about how the code was produced.

## Git safety (required before every commit)

Run in parallel:

- `git status`
- `git diff` (staged and unstaged)
- `git log -1 --oneline` (match recent message style)

Then:

1. Stage only files for the current implementation step.
2. Commit with HEREDOC message (PowerShell-safe quoting on Windows).
3. `git status` after commit to confirm success.

Never commit: `backend/.env`, `*.db`, `plans/`, secrets, local SQLite files.

Never use: `--no-verify`, `git commit --amend` unless all user-rule amend conditions are met.

Never: `git push` unless the user explicitly asks.

## Fix-up commits

If verification fails after a commit:

- Fix the cause in a **new** commit (preferred).
- Amend only when user rules allow and HEAD was created in the same session without push.
